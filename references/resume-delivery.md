# Resume Delivery

## Decide the number of resumes

Create one resume when viable roles share the same responsibility pattern, proof expectations, and core narrative. Create another version only when at least two of these change materially:

- primary responsibility;
- hiring evidence;
- project ordering;
- technical vocabulary;
- business context;
- seniority narrative.

Prefer one to three strong resume families. Do not create one resume per job by default.

## Resume claim gate

Use a candidate fact in a resume only when:

- status is `confirmed` or `evidence_available`, or `evidence_private` with an approved `abstract_only` rendering;
- disclosure is `public` or `abstract_only`;
- the candidate's responsibility is distinguishable from team, author-list, or organization attribution;
- wording does not upgrade participation into leadership, prototype into production, or planned work into completion.

Exclude `planned`, `in_progress`, `unknown`, `disallowed`, and private-only details from completed-experience claims. Ongoing work may appear only in an explicitly current-work section with accurate status.

## Build each bullet

Prefer this evidence order:

1. problem and operating constraint;
2. candidate responsibility and decision;
3. technical action or trade-off;
4. validation method;
5. generalized or public result.

Do not force a metric when none is available. Replace invented numbers with a concrete validation method, comparison, production boundary, reliability action, or artifact.

## Confidential work

Describe system layer, problem class, responsibility boundary, engineering constraint, validation, monitoring, and generalized result. Remove client names, internal dataset details, confidential algorithms, unpublished metrics, topology, endpoints, and identifiers.

Create a public reproduction task when the private work is important but cannot be shown.

## Tailoring

For every resume family, define before drafting:

- target role cluster;
- one-sentence candidate thesis;
- evidence priorities;
- project and publication order;
- terms that must appear naturally;
- facts to omit because they distract or create an unsupported seniority signal;
- hard gaps the resume must not conceal.

Keep personal contact fields as placeholders unless the user supplied them for the final artifact.

## Word delivery

When the user requests `.docx`, generate the document through the available Word/document capability. Render it to page images or PDF and inspect page breaks, headings, bullets, alignment, font substitution, and orphan lines before delivery. Content validation does not replace visual validation.
